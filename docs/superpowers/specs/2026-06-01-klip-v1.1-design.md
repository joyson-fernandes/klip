# Klip v1.1 — Design Spec

Date: 2026-06-01

## Overview

v1.1 rebrands **gifsnap → klip** and adds three new capability surfaces:

1. **Screenshot capture** — a second capture mode that produces a single PNG (no animation, no encoding).
2. **Annotation editor** — a markup window that opens automatically after every screenshot, with 14 tools.
3. **Configurable hotkeys** — both capture hotkeys are user-editable via the settings popover.

The existing GIF capture flow is unchanged in behaviour.

## Feature 1 — Screenshot capture

### Hotkey

Default **⌘⇧2**. Pressing it triggers the existing `RegionSelector` flow. On `regionSelector(_:didSelect:on:)`:

- If the active hotkey was the GIF one → existing behaviour (start recording).
- If the active hotkey was the screenshot one → new code path: capture a single image of the region using `SCStream` with `queueDepth: 1` and one frame, OR via `CGWindowListCreateImage` for instant single-frame capture (no permission re-prompt overhead).

We use **`CGWindowListCreateImage` for screenshots**. It is synchronous, returns a `CGImage` directly, doesn't need a stream lifecycle, and works under the same Screen Recording permission grant. It also bypasses any frame-rate latency.

### Output

The captured `CGImage` is wrapped in `NSImage`, opened in the annotation editor. The user-facing flow is: hotkey → drag → press ↵ or release mouse → editor window opens with the screenshot loaded.

There is no "save and skip editor" option in v1.1. Every screenshot opens the editor — if the user just wants the raw screenshot, they hit ⌘S immediately.

## Feature 2 — Annotation editor

A separate `NSWindow` (not a panel, not full-screen) titled with the capture dimensions.

### Layout

Three rows from top to bottom:

1. **Title bar** (`klip` mark · "Untitled · 1284 × 720" · Cancel · Save). Save is the prominent purple button.
2. **Toolbar** — single horizontal row with 14 tools, grouped by subtle dividers, plus colour swatches + width slider on the right.
3. **Canvas** — a centred view with the screenshot rendered. Tools draw on an overlay layer above the screenshot bitmap.

### Tools (14)

| Group | Tools |
|---|---|
| Selection | Select |
| Shapes | Arrow, Rectangle, Ellipse, Line |
| Content | Pen (freehand), Text, Speech bubble, Step counter (auto-numbering 1, 2, 3…) |
| Effects | Highlight, Blur, Pixelate, Spotlight |
| Crop | Crop (changes canvas extent) |

### Tool style

- **Colour palette**: 8 swatches (red, orange, yellow, green, blue, purple, black, white). Active swatch outlined.
- **Width slider**: 1–20 px. Applies to stroke-based tools (arrow, line, rectangle, ellipse, pen).
- **Text** uses size + colour, no width.
- **Effects** (blur, pixelate, spotlight) have a single radius/intensity slider replacing the width slider when active.

### Interaction

- Click a tool → cursor becomes the relevant icon.
- Click-drag on canvas → creates a new shape committed on mouse-up.
- Select tool → click an existing shape to select; arrow keys nudge; delete key removes; drag handles resize.
- ⌘Z / ⌘⇧Z → undo/redo.
- ⌘S → save (annotated PNG written to save folder + copied to clipboard + window closes + notification).
- Esc → cancel (window closes, nothing saved).

### Persistence

Annotations are not saved as editable layers — once the user hits Save, the final flattened PNG is written. There is no "open existing annotation file to keep editing" flow.

### Architecture

New components:

- `Screenshot` — captures a `CGImage` from a `CGRect` on a given `NSScreen`.
- `EditorWindowController` — owns the editor window's lifecycle.
- `AnnotationCanvas` — `NSView` subclass that renders the screenshot bitmap + a list of `Annotation` model objects.
- `Annotation` — protocol with concrete types per drawn tool: `ArrowAnnotation`, `RectAnnotation`, `EllipseAnnotation`, `LineAnnotation`, `PenAnnotation`, `TextAnnotation`, `SpeechAnnotation`, `StepAnnotation`, `HighlightAnnotation`, `BlurAnnotation`, `PixelateAnnotation`, `SpotlightAnnotation`.
- **Crop** is not an `Annotation`. It mutates the canvas extent itself: the user drags a crop rectangle, presses ↵, and the screenshot bitmap + every existing annotation's coordinates are translated to the new origin. Crop history pushes a single undoable `CropOperation` onto the undo stack alongside annotations.
- `ToolbarView` — SwiftUI view with tool grid + colour swatches + width slider.
- `EditorState` — `ObservableObject` holding selected tool, colour, width, annotation list, undo/redo stacks.
- `AnnotationRenderer` — flattens canvas to PNG `Data` using `NSImage(size:).draw(in:)` + `bitmapImageRepForCachingDisplay`.

## Feature 3 — Configurable hotkeys

### Popover changes

A new **Hotkeys** section added below the Save folder row:

```
HOTKEYS
Screenshot     [⌘ ⇧ 2]   ⓧ
Record GIF     [⌘ ⇧ G]   ⓧ
                 Reset to defaults
```

Each row shows the current combo in a recorder field. Click the field → it shows "Press combo…" and listens for the next key event. Press combo → field updates. Click ⓧ → clears the binding (no hotkey assigned). Reset link restores both defaults.

### Implementation

- A `HotkeyRecorder` SwiftUI view binds to a `KeyCombo?` (struct: `keyCode: UInt32`, `modifiers: UInt32`).
- `SettingsStore` adds two new properties: `screenshotHotkey: KeyCombo?`, `gifHotkey: KeyCombo?` — encoded as JSON in UserDefaults.
- `MenuBarController` is refactored: instead of one hardcoded hotkey, it owns two `RegisteredHotkey` references and a small `HotkeyManager` that registers, unregisters, and rebinds via Carbon when settings change.
- `HotkeyManager` listens to `SettingsStoreObservable` via Combine — any change reregisters the affected hotkey.

### Validation

- The recorder ignores combos with no modifier key (would conflict with normal typing).
- The recorder allows the same combo for both — the user is free to make it confusing.
- We do not detect system-shortcut conflicts. macOS will silently fail to register; we surface that as a notification once on app launch if either hotkey didn't register.

## Out of scope (still)

- Video / MP4 recording
- Cloud upload (Imgur, S3)
- App Store distribution
- Multi-monitor capture spanning displays
- Editing existing image files (only freshly captured screenshots)
- Annotation on GIFs (already decided in brainstorming)

## Migration plan

1. Rename source folders: `gifsnap/` → `klip/`, `gifsnapTests/` → `klipTests/`. Xcode scheme renamed accordingly. `project.yml` target renamed.
2. Bundle ID changes to `com.joyson.klip` — old TCC grants under the old bundle ID become orphaned. Users will be prompted once for Screen Recording on first v1.1 launch. Acceptable since the user base is the developer themselves.
3. State files in the existing medium-sync repo are unaffected (different project).
4. Old `gifsnap.app` removed from `/Applications/` during the install; new `klip.app` takes its place.
